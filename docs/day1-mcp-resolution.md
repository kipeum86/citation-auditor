# Day 1 Korean-law MCP 해상도 스파이크

날짜: 2026-04-21
목적: `korean-law` verifier skill이 claim을 어느 해상도까지 검증할 수 있는지, 어떤 MCP 도구 체인을 써야 하는지 실측 확정.

## TL;DR

| 기능 | 결론 | 함의 |
|---|---|---|
| **조 단위 법령 텍스트 조회** | ✅ 안정적 | `get_law_text(lawId, jo)` 사용 |
| **항 단위 검증** | ✅ 텍스트 파싱으로 가능 | 반환 텍스트에 `①/②/③` 마커 보존 |
| **호 단위 검증** | ✅ 텍스트 파싱으로 가능 | 반환 텍스트에 `1./2./3.` 번호 보존 |
| **주소 단위(parse_jo_code) 항/호** | ❌ 조 단위까지만 | "제15조 제1항 제2호" 입력해도 "001500" 리턴 |
| **법령 약칭 자동변환** | ⚠️ 불안정 | "민법" 검색 시 display≥20 필요 |
| **판례 번호 정확 매칭** | ⚠️ `query` 검색 + 결과 순회로 가능 | `caseNumber` 파라미터 말고 결과 리스트의 `사건번호:` 필드 수동 비교 |
| **판례 제목 기반 쟁점 확인** | ✅ 제한적으로 가능 | `search_precedents` 결과 `TITLE`과 claim 주제를 비교해 명백한 topic mismatch 감지 가능 |
| **판례 전문 조회** | ⚠️ 자주 실패 | `get_precedent_text`가 "not found"로 떨어지는 빈도 높음 |
| **관련 판례 자동 추출** | ⚠️ 노이즈 큼 | `get_article_with_precedents`의 related는 조 번호 substring 언급에 불과, 법리 관련성 아님 |

**Week 3 반영 스코프:** **법령 본문(조/항/호) 검증은 신뢰성 있게 구현 가능.** **판례는 사건번호 존재 확인 + 검색 결과 제목 기준의 명백한 쟁점 불일치 탐지까지 구현 가능.** 다만 **판례 원문 전체를 읽어 세부 판시를 확정하는 자동 검증은 아직 신뢰하기 어렵다.** 세부는 아래 "Verifier skill 권장 로직" 참조.

## Week 3 live E2E addendum

실서비스 E2E에서 `대법원 2023다302036 판결은 확률형 아이템 소비자 보호 의무 위반 시 사업자의 손해배상 책임을 인정한 선례이다` 같은 claim을 넣어보니, 사건번호 자체는 실제로 존재했지만 `search_precedents` 결과 제목은 다음과 같았음:

> 이 사건 확약서, 이 사건 특약사항은 모두 민법 제103조에서 정한 반사회적 법률행위에 해당하여 무효에 해당함

핵심 함의:

- 사건번호만 보면 "존재함"이라서 ❓로 끝나기 쉬움
- 그러나 **검색 결과 제목만으로도 claim의 주제와 판례의 실제 쟁점이 전혀 다름**을 판단할 수 있음
- 따라서 Week 3부터는 precedent protocol을 **"존재 확인 only" → "존재 확인 + title topic mismatch check"**로 상향
- 반대로, 제목과 claim 주제가 어느 정도 겹치거나 애매하면 여전히 `❓`로 남기고 원문 확인을 권장

## 실측 케이스

### 1. `parse_jo_code` — 주소 단위 한계

```
입력: "제103조"           → 출력: "010300"  (AAAABB, AAAA=조, BB=의X)
입력: "제15조 제1항 제2호" → 출력: "001500"  (항/호 정보 소실)
```

**함의:** MCP의 addressable unit은 조 단위. 항/호 매칭은 반드시 조 전체를 받아온 뒤 텍스트 파싱으로 해야 함.

### 2. `get_law_text` — 본문 해상도

**개인정보 보호법 제15조 (항+호 구조):**
```
제15조(개인정보의 수집ㆍ이용)
① 개인정보처리자는 다음 각 호의 어느 하나에 해당하는 경우에는 ...
1.  정보주체의 동의를 받은 경우
2.  법률에 특별한 규정이 있거나 ...
3.  공공기관이 법령 등에서 정하는 ...
4.  정보주체와 체결한 계약을 ...
5.  명백히 정보주체 또는 제3자의 ...
6.  개인정보처리자의 정당한 이익을 ...
7.  공중위생 등 공공의 안전과 안녕을 ...
② 개인정보처리자는 제1항제1호에 따른 동의를 받을 때에는 ...
1.  개인정보의 수집ㆍ이용 목적
2.  수집하려는 개인정보의 항목
...
③ 개인정보처리자는 당초 수집 목적과 ...
```

**민법 제108조 (항만):**
```
제108조(통정한 허위의 의사표시)
①상대방과 통정한 허위의 의사표시는 무효로 한다.
②전항의 의사표시의 무효는 선의의 제삼자에게 대항하지 못한다.
```

**민법 제103조 (단문):**
```
제103조 반사회질서의 법률행위
선량한 풍속 기타 사회질서에 위반한 사항을 내용으로 하는 법률행위는 무효로 한다.
```

**함의:**
- 항 마커 `①②③...⑩⑪⑫...` 유니코드 circled digit 보존.
- 호 마커 `1.  ` (공백 2개 포함) 보존.
- 항 없는 단문 조문은 마커 없이 본문만 반환 — 일관된 포맷.
- **한 번의 `get_law_text(lawId, jo)` 호출로 조 안의 모든 항/호 텍스트를 확보할 수 있음.**

### 3. `search_law` — 약칭/기본어 이슈

```
search_law(query="민법", display=3)  → 난민법, 난민법 시행령, 난민법 시행규칙
search_law(query="민법", display=20) → ..., 4번째에 "민법 (lawId=001706)" 등장
```

**함의:**
- search_law는 substring 매칭 + 공포일/알파벳 랭킹. 짧은 이름의 핵심 법령이 뒤로 밀림.
- 약칭 자동변환이 문서엔 있지만 실측상 "민법" → 민법 lawId 바로 매핑 안 됨.
- **Skill 전략:** 번들된 `lawId` 룩업 테이블을 skill 내부에 둔다. 민법/형법/상법/민사소송법/형사소송법/개인정보 보호법 등 자주 쓰는 법령 20-30개는 hardcode.
- Fallback: search_law with display≥20 → 결과 순회하며 `구분: 법률` 필터 + 쿼리 정확 일치 선택.

### 4. `search_precedents` — 판례 매칭 신뢰도

**캐이스 A: 존재하지 않는 사건번호**
```
search_precedents(caseNumber="2019다249428", display=3)
  → 결과 4건 반환. 전부 다른 사건번호(2024다211427, 2023다304650, 2022다312265).
  → caseNumber 필드는 exact match가 아니라 유사 주제 추천 느낌.
```

**케이스 B: 존재하는 사건번호 (키워드 검색으로 역확인)**
```
search_precedents(query="2023다302036", display=3)
  → 결과 1건: [245007] 사건번호: 대법원-2023-다-302036
  → 키워드 쿼리로 검색하면 정확 매칭 가능. 단, 결과 포맷의 사건번호에 dash 삽입됨.
```

**함의:**
- caseNumber 파라미터 단독으로는 "존재 여부 확인" 불가능. Claim의 판례 번호 검증 경로:
  1. `search_precedents(query=<claim의 판례번호>)` — `2023다302036` 같은 형태 그대로 키워드 쿼리.
  2. 결과 리스트 순회하며 각 항목의 `사건번호:` 필드를 **dash 제거 후 정규화**해 claim과 정확 비교.
  3. 매칭되면 "존재 확인" verdict. 안 되면 ❓.

### 5. `get_precedent_text` — 전문 조회 실패율

```
get_precedent_text(id=330456)  → "Precedent not found or invalid response format"
get_precedent_text(id=245007)  → "Precedent not found or invalid response format"
```

두 ID 모두 방금 search_precedents에서 반환된 값이었음에도 전문 조회 실패.

**함의:**
- **판례 전문 비교 경로는 여전히 신뢰할 수 없음.** 실제 전문으로 claim의 판시 내용을 비교 검증하는 것은 불가능한 경우가 많음.
- 다만 `search_precedents`의 **결과 제목(TITLE)** 은 신뢰 가능한 요약 힌트로 볼 수 있어, 사건번호가 맞더라도 claim 주제가 명백히 다른 경우는 ⚠️까지 올릴 수 있음.
- 판례 내용(판시사항/판결요지)까지 세부 비교해야 하는 claim은 제목만으로 충분치 않으므로 ❓로 남기고 사람이 직접 원문 확인.

### 6. `get_article_with_precedents` — 관련 판례 노이즈

민법 제103조 조회 시 "관련 판례" 5건 리턴:
- #500759: 제108조 제2항 관련 (노이즈)
- #245007: 제103조 관련 (적합)
- #347904: 제830조 제1항 관련 (노이즈)
- #211030: 제29조 제1항 관련 (노이즈)

4건 중 1건만 실제 제103조 법리 관련.

**함의:** "관련 판례" 목록은 조문 번호가 판례 텍스트에 언급된 경우를 긁어온 것. **verifier가 이 목록을 그대로 사용하면 안 됨.** 필요하면 추가 필터링(판시사항에 실제 조문 번호가 등장하는지 확인) — 그러나 v1 스코프 초과. v1에서는 `includePrecedents=false`로 쓰거나 아예 `get_law_text`만 사용 권장.

## Verifier skill 권장 로직 (Week 3 반영)

### 법령 claim 검증 (조 단위)

1. Claim 파싱: 법령명, 조 번호 추출. 예: `민법 제103조` → `law="민법", jo="제103조"`.
2. 법령 lawId 해결:
   - 1차: skill 내 hardcoded 룩업 테이블 (민법=001706, 형법=<TBD>, 개인정보 보호법=011357, ...).
   - 2차 fallback: `search_law(query=law, display=20)` → `구분: 법률` + 쿼리 정확 일치 필터.
   - 실패: ❓ "법령을 식별하지 못함".
3. 조문 텍스트 로드: `get_law_text(lawId, jo)`.
4. Claim이 조 단위 주장이면 본문 전체 vs claim 비교 → verdict.
5. Claim이 항/호 단위 주장이면 추가 파싱:
   - 항 마커 `①②③...` 로 split → 해당 항 텍스트 추출.
   - 호 마커 `숫자.` 로 split → 해당 호 텍스트 추출.
   - 해당 항/호 없으면 ⚠️ "조문에 해당 항/호 없음".

### 판례 claim 검증 (사건번호 확인 + 제목 기반 쟁점 점검)

1. Claim 파싱: 사건번호 추출. 예: `대법원 2023다302036` → `caseNumber="2023다302036"`.
2. 검색: `search_precedents(query=caseNumber, display=5)` — **query로, caseNumber 파라미터 아님**.
3. 결과 순회: 각 항목의 `사건번호:` 필드를 dash 제거 후 claim과 정확 비교.
4. 일치하는 항목을 찾으면 그 결과의 `TITLE`과 검색 결과 ID를 함께 보관.
5. Claim이 단순 존재 주장(`... 판결이 존재한다`)이면 ✅ "사건번호가 확인되었습니다."
6. Claim이 판례의 주제/쟁점/법리를 서술하면, claim의 주제 키워드와 `TITLE`의 주제 키워드를 비교.
7. 주제가 **명백히 다르면** ⚠️ "사건번호는 확인되지만, 판례의 실제 쟁점이 주장과 다릅니다."
8. 주제가 어느 정도 겹치거나 애매하면 ❓ "사건번호는 확인됩니다. 판례의 정확한 판시 내용은 원문 확인을 권장합니다."
9. 매칭 없으면 ⚠️ "해당 사건번호를 확인할 수 없습니다."
10. **판례 내용(판시사항/판결요지)까지 전문 비교는 자동화하지 않음.** `get_precedent_text` 실패율 높음 + 제목만으로 세부 법리까지 확정하기 어렵기 때문.

### 이연 (v1.x)

- 판례 본문 비교 검증 — `get_precedent_text` 안정화 대기.
- 관련 판례 자동 필터링 — 조문-판례 법리 관련성 스코어링 필요.
- 시행령/시행규칙 연쇄 검증 — `get_three_tier` 활용해 위임 관계까지 검증.
- 법령 개정 히스토리 검증 — `get_law_history` 활용해 claim의 시점 정확성 판정.

## 운영 메모 — speculation boundary

이 스파이크와 별개로, 실제 `/audit` 운용에서는 **예측/전망/의견/전언** 문장을 claim extractor가 기본적으로 잡지 않는 것이 맞다. 예:

- `업계 관계자에 따르면 규제가 완화될 전망이다`
- `향후 판례가 바뀔 가능성이 있다`

이런 문장은 공적 원문과 즉시 대조할 수 있는 사실 주장으로 보기 어렵다. citation-auditor는 **검증 가능한 사실 주장과 인용** 중심으로 감사를 수행한다.
